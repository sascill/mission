# Development Workflow

## 목적
- 기능 구현 전에 작업 흐름을 먼저 정리해서 이슈, 브랜치, PR을 일관된 방식으로 관리한다.
- 작업 단위마다 배경, 완료 조건, 예상 시간, 기술 선택 이유를 기록해 개발 과정을 추적 가능하게 만든다.

## 작업 흐름
1. GitHub Wiki의 Backlog에서 작업 후보를 정리한다.
2. Backlog 항목을 기준으로 GitHub Issue를 생성한다.
3. 이슈 번호를 기준으로 브랜치를 만든다.
4. 브랜치에서 구현 또는 문서 작업을 진행한다.
5. 작업 내용을 정리한 뒤 Pull Request를 생성한다.
6. 리뷰 후 main 브랜치에 병합한다.

## 브랜치 규칙
- 기능 작업: `featrue/<issue-number>-<short-title>`
- 문서 작업: `docs/<issue-number>-<short-title>`
- 하나의 브랜치에서는 하나의 이슈만 다룬다.

## Issue 작성 규칙
- `.github/ISSUE_TEMPLATE.md` 템플릿을 기준으로 작성한다.
- 적절한 label을 지정한다.
- 아래 항목을 빠짐없이 작성한다.
  - 배경
  - 작업 내용
  - 완료 조건
  - 예상 소요 시간

## PR 작성 규칙
- `.github/pull_request_template.md` 템플릿을 기준으로 작성한다.
- 아래 항목을 빠짐없이 작성한다.
  - 작업 내용
  - 기술 선택 이유
  - 개발 중 이슈
  - 테스트
  - Related Issue
- 관련 이슈는 `closes #번호` 형식으로 연결한다.

## 운영 원칙
- 기능 구현 전에 필요한 작업 관리 항목을 먼저 등록한다.
- 작업 중에는 이슈 범위를 벗어나지 않는다.
- UI와 상태가 함께 필요한 작업은 상태 소유 컴포넌트를 먼저 정하고 구현한다.
